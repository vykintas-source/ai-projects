Data Infrastructure & Synchronisation Infrastructure
+

Decide and document where data lives across all projects and how it works for the team.

Three Storage Levels

  1. Cloud (e.g. Google Drive): accessible from anywhere, automatic sync, suitable for shared files
  2. Office server / shared drive: physical hard drive connected to the server, configured as a virtual drive, accessible to all office members over the network
  3. Local machine: each team member holds their own portion locally

Questions to Resolve

  • What lives where: which data goes to cloud, which to server, which stays local
  • How much data needs to be in the cloud vs on local machines
  • How Claude accesses data from each level: via cloud, local folders, or both
  • If multiple members work on the same file, shared access is needed: Google Drive, server, or both

Sync Hygiene

  • Define sync rules per project
  • When and how files are synced between all three levels
  • How to avoid conflicts and data loss

02
24/7 Agent Infrastructure Infrastructure
+

The agency needs a solution for running agents and automated tasks around the clock without relying on a local machine being always on.

What Needs to Run Continuously

  • Recurring agent tasks
  • Database operations and syncing
  • Knowledge base maintenance and updates
  • Scheduled jobs of all kinds

Starting Point

  • A dedicated always-on office machine is an acceptable starting point
  • As the number of agents grows and more agency processes depend on AI, reliance on a single machine becomes a critical vulnerability
  • At some point this must migrate to cloud servers or a serious dedicated server machine
  • This transition needs to be planned in advance, not reactive

Core Questions to Answer

  • What kinds of recurring tasks exist across all projects, and what are their requirements (frequency, compute, data access)
  • What cloud server options exist and what fits the agency's scale and budget
  • Who manages and maintains the infrastructure, and what level of technical overhead is acceptable
  • How do agents access local data and tools from a remote server
  • How does this connect to the data infrastructure project (#1): cloud, office server, and local machines
  • What is the cost model: VPS, managed cloud (AWS/GCP/Azure), or something else
  • Security: who has access, how are credentials managed
  • Reliability and uptime requirements
  • Monitoring: how do we know when something breaks or fails silently

This project needs to map all recurring tasks across all other projects first, then design the infrastructure around actual needs rather than assumptions.

Amazon Seller Support Automation Automation
+

Problem: a single support case takes ~2 hours, most of it wasted time.

Intake

  • Three fields: Facts / Problem / Target
  • If a precedent exists: the person takes an existing snippet, edits it if needed, and uses it
  • If not: fills out the full intake from scratch
  • Over time the snippet library grows and intake gets faster

Agent 1 — Operator (Claude in Chrome)

  • Runs the actual support chat
  • Follows playbooks
  • If stuck, escalates to a human via Slack
  • Never self-modifies, only reads updated instructions

Agent 2 — Analyst (Claude Code)

  • Works after the conversation ends
  • Analyses transcripts
  • Classifies cases, extracts patterns
  • Proposes playbook updates, human approves
  • Grows the knowledge base, compiles new cases

Slack

  • Escalation channel
  • Audit log

Knowledge Base

  • Project's own knowledge base
  • Can be connected to the Amazon Knowledge Base
  • Playbooks, raw transcripts, processed case logs, snippets, learnings
  • Central case index with controlled taxonomy for searchability

Operations & Tech Stack

  • Claude in Chrome so actions appear human and are not blocked by Amazon
  • No custom code required
  • Built on: Claude in Chrome, Claude Code, Projects, Slack MCP
