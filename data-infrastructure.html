Problem Statement

Building a shared agency-wide knowledge base and file infrastructure. The system must work for the full team — office and remote — from day one, not just a few accounts. Initial deployment starts with 3 Claude accounts and 3 machines, but the architecture is designed for the full agency from the start.

Requirements

  • Accessible from any machine, anywhere — office, remote, any country
  • Works via Claude web app (claude.ai) — the primary interface across the team
  • Works via Claude Desktop on machines where it is installed
  • Works on phone and any browser
  • Full team access: currently 20+ users on Claude Team plan, scaling to 100+ users
  • Half the team works remotely — remote access is not optional, it is the baseline
Local filesystem MCP was ruled out — it is machine-specific and incompatible with the web app.

Confirmed Architecture

Layer 1 — Storage: Google Drive (dedicated Workspace user)

  • A dedicated user created within the agency's existing Google Workspace — e.g. ai@youragency.com
  • No new Workspace subscription needed — one additional user seat on the existing plan
  • Sits inside the agency Workspace, inheriting enterprise-grade API quotas, Shared Drive capabilities, admin controls, and Google's uptime SLA
  • Scales to 100+ users without hitting rate limits that would affect a personal Google account
  • Native folder-level permission control — ready for role-based access management at any scale
  • Google Drive for Desktop installed on working machines mounts the Drive as a local folder (e.g. G:\My Drive\). Claude Desktop's filesystem MCP then treats it like any local directory — full read/write, no difference from a local drive.
  • Offline edits sync when connection is restored. Simultaneous edits on two machines produce conflict copies (standard behavior, same as Dropbox).

What Lives Here

  • CLAUDE.md context files
  • Client project files
  • Zervox / COSMO research
  • Databases, datasets
  • Everything the team needs

Layer 2 — Remote MCP Server: Railway

  • A small hosted service wrapping the Google Drive API with full read/write
  • Exposes a URL that all Claude accounts connect to via MCP settings
  • Already familiar with Railway — no new platform to learn
  • Managed platform: Railway handles uptime, no server admin required
  • Estimated cost: ~$5–15/mo depending on usage
  • Scales to 100+ users: same server URL, access control per user at server level
  • Must be built production-quality from the start — connection pooling, caching, rate limit handling

What This Enables

  • One shared knowledge base for the entire agency — the core value. Currently not possible with any existing setup.
  • Office staff and remote workers access the same database equally — no distinction between being in the office or working from anywhere in the world.
  • Access via any path: Claude web app, Claude Desktop + Drive for Desktop local mount, phone, any browser anywhere.
  • Claude web app gets full read/write to Drive — not just read-only.
  • Any Claude account connected to the server gets access instantly — no per-machine setup beyond a browser.
  • Future: per-user and per-team folder access restrictions enforced at MCP server level.

Layer 3 — Client-Facing Reports: Vercel or Netlify (free tier)

  • Weekly reports and analysis published as HTML files
  • Public URLs with unguessable link hashes (industry-standard approach — same as Google Docs link sharing, Loom, Notion public pages)
  • No client accounts needed — they just open a link
  • No cost (free tier sufficient)
  • Access model: link = access. If confidentiality is critical, an additional layer can be added later, but for weekly analysis this is sufficient

Access Control Model

Current — Starting Point (3 accounts)

  • 3 Claude accounts connect to same Railway MCP URL
  • Google Drive folder structure determines what's accessible
  • No per-user restriction yet — full shared access among initial users

Near-Term — Full Agency

  • All team members connect via their Claude accounts to the same MCP URL
  • No machine setup required — browser is enough for remote staff
  • Google Drive folder permissions control who sees what at the folder level

Future — 100+ Users, Role-Based Access

  • MCP server enforces per-user and per-role folder access
  • Simple users see only what they need — client files, their own projects
  • Senior staff and admins see everything
  • Architecture already supports this — no rebuild needed

Key Decisions Log

  • Google Drive chosen over Notion (not a real filesystem), S3 (no native UI), Dropbox (no advantage)
  • Dedicated Workspace user (ai@youragency.com) chosen over personal Google account — enterprise quotas, scales to 100+ users, no new Workspace subscription needed
  • Railway chosen because already in use; MCP server must be built production-quality from the start (not a prototype) to handle 100+ concurrent users
  • Unguessable link access for client reports — sufficient for weekly analysis use case
  • Managed platforms preferred throughout — no self-managed servers
  • Scale target: 100+ users. Architecture supports this without a rebuild.
